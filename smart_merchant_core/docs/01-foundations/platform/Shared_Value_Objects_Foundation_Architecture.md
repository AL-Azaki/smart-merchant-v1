# Shared Value Objects Foundation Architecture

**Status:** APPROVED  
**Version:** 1.0  
**State:** FROZEN  

---

## 1. Purpose
تمثل هذه الوثيقة الدستور المعماري المشترك (Shared Foundation) لجميع الـ Value Objects التي يتم استخدامها كعناصر متكررة عبر مختلف النطاقات (Domains) في نظام Smart Merchant ERP. تهدف الوثيقة إلى توحيد المفاهيم والقواعد الخاصة بهذه الكيانات لضمان سلوك متناسق، آمن، ومستقل عن أي منطق عمل (Business Logic) خاص بنطاق معين.

---

## 2. Scope
يغطي هذا الدستور المعماري المبادئ والقواعد الأساسية الخاصة بالـ Value Objects المشتركة، ولا يمثل نطاقاً مستقلاً بحد ذاته. جميع النطاقات (Finance, Sales, Inventory, Purchasing, إلخ) ملزمة باتباع هذه القواعد عند استخدام أو تعريف هذه العناصر.

---

## 3. Shared Value Objects
تتضمن القائمة التالية الـ Value Objects المشتركة ومسؤولية كل منها:
- **Money:** يمثل المبالغ المالية. مسؤول عن احتواء القيمة الرقمية وضمان دقتها الرياضية (Decimal Precision) ومنع القيم غير المنطقية.
- **Quantity:** يمثل الكميات المخزنية أو المباعة. مسؤول عن ضمان القيم الموجبة أو السالبة بناءً على السياق، ودقة الأرقام العشرية.
- **Percentage:** يمثل النسب المئوية (مثل نسبة الخصم). مسؤول عن ضمان بقاء القيمة الرياضية ضمن النطاق المعقول (عادة بين 0 و 100).
- **Exchange Rate:** يمثل سعر صرف العملة. مسؤول عن دقة الأرقام العشرية العالية (غالباً 8 خانات عشرية) ومنع القيم الصفرية أو السالبة.
- **Currency Snapshot:** يجمع بين معرّف العملة وسعر الصرف في لحظة زمنية محددة. مسؤول عن تجميد الحالة للاستخدام في المستندات.
- **Document Number:** يمثل الأرقام المرجعية للمستندات (مثل الفواتير والقيود). مسؤول عن صحة التنسيق الهيكلي (Format) وطول النص.
- **Date Range:** يمثل فترة زمنية (تاريخ بداية وتاريخ نهاية). مسؤول عن ضمان أن تاريخ البداية لا يتجاوز تاريخ النهاية.
- **Address:** يمثل العنوان الجغرافي (مثل الشارع، المدينة، الرمز البريدي).
- **Contact Info:** يمثل معلومات الاتصال المجمعة لأي طرف في النظام.
- **Email:** مسؤول عن التحقق من الصيغة الصحيحة للبريد الإلكتروني (Format Validation).
- **Phone Number:** مسؤول عن التحقق من التنسيق الدولي أو المحلي لرقم الهاتف.
- **Tax Rate:** يمثل نسبة الضريبة المطبقة. مسؤول عن القيم الرياضية الخاصة بالضرائب.

---

## 4. Immutability Rules
- جميع الـ Value Objects في النظام هي كائنات غير قابلة للتغيير المطلق (Strictly Immutable).
- لا يُسمح بتوفير أي دوال (Setters) لتعديل الحالة الداخلية للكائن بعد إنشائه.
- أي عملية تعديل رياضية أو نصية على الـ Value Object يجب أن تُنتج وتُرجع (كائناً جديداً - New Instance) يحمل القيمة المعدلة.

---

## 5. Validation Principles
- كل Value Object مسؤول حصرياً عن التحقق من صحة بياناته الداخلية فقط (Self-Validation).
- تقتصر عملية التحقق (Validation) على الصيغة (Format)، والنطاق الرياضي (Ranges)، والقواعد الهيكلية الأساسية.
- **يُمنع منعاً باتاً** احتواء الـ Value Object على أي Business Rules معقدة، أو استعلامات لقاعدة البيانات، أو تواصل مع Domain Services للتحقق من صحة القيمة.

---

## 6. Equality Rules
- لا تمتلك الـ Value Objects أي هوية مميزة (No Identity / No UUID).
- تعتمد عملية المقارنة بين كائنين من نفس النوع على **المقارنة بالقيمة (Value Equality)** فقط.
- يعتبر الكائنان متطابقين إذا، وفقط إذا، تطابقت جميع الخصائص الداخلية (Attributes) بينهما تماماً، بغض النظر عن عنوان الذاكرة (Reference).

---

## 7. Serialization Rules
- يجب أن تدعم جميع الـ Value Objects التحويل السلس من وإلى الأنواع الأساسية (Primitives) أو المصفوفات (Arrays).
- هذه القاعدة ضرورية لضمان قدرة طبقات الـ Infrastructure على تخزين الـ Value Object في قاعدة البيانات، وقدرة طبقات الـ Presentation على عرضها بصيغة JSON.

---

## 8. Ownership Rules
- الـ Value Objects لا تمتلك دورة حياة مستقلة (No Independent Lifecycle).
- تُعتبر هذه الكائنات مملوكة بالكامل للكيانات (Entities) أو الكيانات الجذرية (Aggregate Roots) التي تحتويها.
- تزول الـ Value Objects وتُحذف من الذاكرة أو قاعدة البيانات بمجرد زوال أو حذف الكيان المالك لها.

---

## 9. Dependencies
يعتبر هذا الدستور طبقة أساسية ولا يعتمد على أي وثيقة معمارية لـ Domain معين. على العكس، فإن الوثائق المعمارية التالية تستفيد منه وتحترمه:

## 10. Out Of Scope
يخرج عن نطاق مسؤولية الـ Value Objects وتعتبر المكونات التالية **Out Of Scope**:
- منطق الأعمال المتقدم (Business Logic).
- خدمات النطاق (Domain Services).
- مستودعات البيانات (Repositories).
- وحدات التحكم (Controllers) ونقاط الاتصال (API).
- التفاصيل التقنية الخاصة بالبنية التحتية (Infrastructure) أو قواعد البيانات (ORM).
